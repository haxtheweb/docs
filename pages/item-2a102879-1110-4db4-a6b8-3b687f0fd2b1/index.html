Organization-wide skeletons (shared templates)

HAXiam supports organization-wide skeletons by injecting a shared directory into HAXcms skeleton discovery. This lets administrators provide approved templates for everyone while preserving private user templates.

Skeleton resolution order is:

  1. User-private skeletons: users/<username>/_config/user/skeletons
  2. User instance skeletons: users/<username>/_config/skeletons
  3. Organization shared skeletons: _iamConfig/skeletons
  4. HAXcms core skeletons: cores/HAXcms-1.x.x/system/coreConfig/skeletons

Because of this precedence, a user can still save personal skeletons without publishing them to everyone, while organization-provided skeletons are available globally.

Version control workflow for shared skeletons

The recommended approach is to manage _iamConfig/skeletons as its own Git repository (or submodule) on the server. This allows teams to commit and pull new .json skeleton definitions without putting sensitive _iamConfig files into source control.

  1. Create a repository for shared skeleton files (example: haxiam-skeletons).
  2. Clone that repository into _iamConfig/skeletons in the HAXiam deployment.
  3. Add or update skeleton files as *.json.
  4. Commit and push changes from your management workflow.
  5. On each HAXiam deployment, run git pull inside _iamConfig/skeletons to apply updates for all users.

If you choose to track skeletons directly in the HAXiam repository, update .gitignore carefully so only _iamConfig/skeletons is versioned and secrets in _iamConfig remain ignored.

Operational notes

  • Keep write access to _iamConfig/skeletons restricted to administrators.
  • Treat user-created skeletons as private by default; they remain under each user's own _config space.
  • Use clear machine names and metadata in each skeleton JSON to improve template picker usability.

Vanity domains

HAXiam allows for empowering users to create sites but in order to ensure they don't step on each other's namespaces, addresses can get kind of long. This is by design to support a multi-tenant environment for building the site, but often is annoying when sharing addresses out to others. That's where vanity domains come in, though they require server configuration.

  • Create a valid Apache record for the domain in a VirtualHost block as to where in the file system to point for this domain
  • Include the following environmental variable in the VirtualHost definition: SetEnv HAXSITE_BASE_URL /
  • Where / is the base path seen in the URL when accessing the root of the site. / will be the most common for this when doing something like example.com pointing to a site within a HAXiam instance

The HAXSITE_BASE_URL variable is required to help the front-end resolve where the <base> tag is going to be resolving to. This configuration allows HAXiam to manage the path for viewing and authorship, while at the same time helping the PWA nature of HAXsite still be able to resolve where the files are for the site no matter the URL.

Authentication providers / Customization

Both HAXiam as well as HAXcms can have customization applied to HAXiam's configure directory in order to ensure maintainable upgrades of both the core platform as well as HAXcms which it is managing a copy of.
